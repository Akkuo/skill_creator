# Skill Creator Version History

此文件專門紀錄 `skill-creator` 生態圈中所有技能範本 (Templates)、自動化腳本 (Scripts) 與核心 SOP (`skill.md`) 的迭代歷程與踩坑經驗。
獨立出此檔的原因：**避免日漸膨脹的更新記錄與 Prompt 掛載在一起，造成無謂的 Token 浪費與 AI 的上下文遺忘 (Lost in the Middle) 現象。**

---

## 📅 2024-04-10 (Game Deconstructor Hub & KB 導入)

### 🚀 核心目標
**將原本只在終端機運行的腳本，升級成一個專業的本機網頁工作站 (Game Deconstructor Hub)，並導入「少樣本提示 (Few-Shot Prompting)」機制來優化 AI 產出品質。**

### 🛠️ 變更項目
- **[升級] Web 化操作介面 (`templates/dashboard.html` & `app.py`)**：
  - 放棄終端機黑畫面，建立一個深色主題、高度玻璃擬物化 (Glassmorphism) 的專業儀表板。
  - 左側提供歷史知識庫清單，中區支援 Markdown 編輯與 Mermaid 流程圖即時渲染預覽，右側內建生成、儲存與匯出控制項。
  - 取代了早期陽春的 `web_tool.py`。
  
- **[新增] 知識庫 (Knowledge Base) 與 Few-Shot 迭代機制**：
  - 建立 `data/knowledge/` 本機知識庫目錄。所有拆解好的優質規格書，皆可儲存於此。
  - 後端 `app.py` 串接 Gemini API 時，會自動讀取最新兩份知識庫文章，作為 Prompt 的 Few-Shot 範例。
  - **此架構目的**：隨著使用者手動精修並儲存的案例變多，AI 的產出風格、深度、長度，以及 Mermaid 的正確性會自動朝著「高標」收斂，實現自動品質進化。

- **[安全] 環境變數隔離 (`.env`)**：
  - 將 Gemini API Key (`GEMINI_API_KEY`) 與模型設定 (`GEMINI_MODEL`) 抽離至 `.env` 檔，防止憑證外流。

## 📅 2024-04-10 (Automated Word Pipeline 迭代)

### 🚀 核心目標
**將遊戲企劃規格書 (Game PRD) 從純文字拆解，工業化升級為「全自動圖文 Word 產出流水線」。**

### 🛠️ 變更項目
- **[重構] `企劃規格書.md` 結構優化**：
  - 發現 AI 與人腦的邏輯盲點，將原本混雜的「規則與玩法」強制拆分為 `2. 玩法概要 (Core Gameplay)` 與 `3. 遊戲流程 (Game Rules & Constraints)` 兩個層級，大幅增強邏輯拆解的細緻度。
  
- **[新增] `build_docx.py` (自動化 Word 打包腳本)**：
  - 利用 Python 掃描 Markdown，並串接 `mermaid.ink` API 自動渲染為手繪風格圖片。
  - 接合開源工具 Pandoc，自動編譯出層級分明的 `.docx`，實現產出零手工。

- **[優化] 排版與防呆機制 (防身首異處)**：
  - **Alt-Text 剔除**：移除 Markdown 中 Mermaid 的 `![流程圖]` 代碼，解決 Pandoc 在 Word 底下擅自生成預設圖說 (Caption) 的問題。
  - **OpenXML 強制換頁**：Python 腳本層強制在每張大圖表前方寫入原生 `<w:br w:type="page"/>` 換頁符號，確保後方產生出來的流程圖永遠從漂亮的新反頁開始。
  - **樣式繼承設定**：腳本支援讀取同目錄下的 `word_template.docx` 來繼承微軟的段落樣式（例如勾選「與下段同頁(Keep with next)」），封殺未來文字排版跑掉的可能性。

- **[SOP 升級] `skill.md` 規則收斂**：
  - 新增這份獨立的 `CHANGELOG.md` 規則，強制所有未來踩坑或升級歷史，不可寫入 Skill 本體，以最大化節約 Token 與防止 Context Caching 失效。

## 📅 2026-04-10 (企劃規格書模板品質升級 v2)

### 🚀 核心目標
**根據《Pixel Flow》實機逆向拆解報告，強化 `企劃規格書.md` 的企劃深度，讓 AI 產出的規格書從「概要描述」升級為「可執行的工程設計文件」。**

### 🛠️ 變更項目
- **[A] 核心循環改為心流四步驟 (Observe / Decision / Execute / Goal)**：
  - 原版 Core Loop 只有單向流程描述，無法讓工程師與 UI 設計師理解「玩家在做什麼決策」。
  - 新版強制要求 AI 拆解出微觀的 4 步驟心流循環，貼近實際開發規格。

- **[B] 新增「核心場景物件 (Entities)」章節**：
  - 原版直接跳到操作規則，導致描述抽象（例：「棋盤上有東西」沒寫清楚是什麼）。
  - 新版要求 AI 先定義畫面上所有「存在的物件」（棋盤、罐子、傳送帶等），確保後續規則有清楚的主詞。

- **[C] 難度設計升級為「策略深度與難度痛點 (Strategy & Constraints)」**：
  - 原版難度描述只有數值遞增（速度加快、血量降低）。
  - 新版要求分析難度來源是「空間規劃順序」、「容量資源管理」還是「時機反應」，對應真實的 DDA (動態難度) 設計邏輯。

- **[D] 新增「Game Juice (爽感回饋)」章節**：
  - 休閒遊戲的留存率由「爽感」驅動。原版模板完全缺乏這個面向。
  - 新版要求明確寫出「什麼操作觸發什麼視覺/音效回饋」，為 VFX 與音頻設計師提供具體規格。

- **[E] 引入「三層核心循環法」：新增局外進度 (Secondary Loop)**：
  - 引進業界標準的三階層 Core Loop 拆解法，於 `2-C` 新增 Meta-game 循環。
  - 補足原本企劃書中「玩家通關後拿到資源可以幹嘛？」的盲點，涵蓋資源產出、消耗與長期目標，協助抓對產品的長線留存邏輯。

- **[F] 高度嚴謹的「證據推論原則」寫入底層 Prompt**：
  - 於 `app.py` 系統底層新增嚴格鐵律：「僅依據視覺證據或明確描述回答，嚴禁猜測隱藏邏輯，除非有連續三次以上的重複現象」。徹底封死 AI 針對未明言之數值的擅自腦補行為。

- **[G] 移除非必要工具 (Game Deconstructor Hub 退場)**：
  - 將剛剛建立的 `app.py` 與前端介面物理刪除，回歸最簡化工作流（單純依賴 `skill.md` 與 `企劃規格書.md`）。
  - **策略轉向**：遵循 MVP 原則，先確保 AI 產出的規格書在對話中能達到 100% 預期，再來考慮打包成 Web 自動化工具，避免提早最佳化造成失焦。
